# Constitution — README Portfolio Improvement

**Version:** 1.0 | **Status:** Vigente

---

## Article I — Portfolio as Public Face
Every public repository README is a first impression. It must answer "what is this?" in under 15 words and "how do I use it?" in under 200 words.

## Article II — Design Quality Matches Content Quality
A capability repo (laecon, ladesign, lan8n) with a stub README undermines the ecosystem's credibility. The README's design must reflect the capability's maturity.

## Article III — Bilingual When Appropriate
Portuguese-only repos (logistica-me, pizzarias-sp-2026) get English headers at the top to maximize international reach. Portuguese content stays; English framing is added.

## Article IV — Visual Proof Over Textual Description
Screenshots, GIFs, Mermaid diagrams, and code output examples are worth 10x their weight in text. Every project README shows, not just tells.

## Article V — Consistent Badge System
All repos share a minimum badge set: Python version, License, Status (BASIC/STABLE/MATURE). This creates portfolio-wide visual coherence.

## Article VI — ASCII Art Is Transitional
Mermaid renders natively on GitHub, is maintainable, and supports dark mode. ASCII architecture diagrams are replaced with Mermaid unless the ASCII is genuinely superior.

## Article VII — LAOS Is the Gold Standard
The `laos` README (20/20) is the exemplar. Other READMEs aspire to its patterns (inline HTML/CSS, SVG emblem, visual tables, numbered workflows) but adapt to their domain.

## Article VIII — No Wall of Text
READMEs over 300 lines get collapsible `<details>` sections, TL;DR headers, or visual breaks. Scannability > completeness in the first viewport.

## Article IX — Contributing and License Are Non-Optional
Every public repo has a contributing section and a license reference. No exceptions.

---

## Scope

- 15 public repositories from `github.com/laurentaf`
- README.md files only
- Badge system, visual hierarchy, content completeness, bilingual headers
- Reusable template for future READMEs

## Non-Goals

- Private repos (emprestisa, career-ops, lacareerops, previsao-concursos, brasfoot-poc, template-base, lopes-melo-telecom, personal)
- Code implementation or feature changes
- Repository structure reorganization
- CI/CD pipeline changes
- Modifying the LAOS README itself (it's the exemplar, not the target)
